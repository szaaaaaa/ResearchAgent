# Dynamic Research OS 产品需求文档
> 日期：2026-03-12
> 状态：Draft v1

## 1. 概述

Dynamic Research OS 是一个由 `planner` 驱动的多智能体研究系统。

系统保留 `planner + 6 agents`，但移除旧的固定执行流水线。`planner` 会根据用户请求，动态决定是否调用以下 6 个执行角色中的一部分：

- `conductor`
- `researcher`
- `experimenter`
- `analyst`
- `writer`
- `reviewer`

默认执行规则为：

`role -> skill -> tool`

其中：

- `planner` 只做局部规划，不做一次性全局固定规划
- `skill` 是可复用的任务单元
- `tool` 是最底层执行单元，优先指 `MCP` 工具和受控执行后端
- `tool` 在启动时从已配置的 `MCP` 服务中发现，并通过模块化的 `ToolGateway` 暴露给运行时

## 2. 问题背景

当前系统虽然已经出现了部分“动态路由”的外观，但真实执行仍然大量依赖旧的固定阶段逻辑与硬编码角色行为。

这会带来几个问题：

- `planner` 还不是唯一真实的流程决策者
- 角色仍部分依赖硬编码，而不是由规划器调度的执行单元
- 系统很难通过用户自定义 `skill` 扩展
- `review` 行为仍受旧架构约束，而不是由 `planner` 决定
- 执行失败后的恢复路径还不够 contract 化

## 3. 产品目标

构建一个新的研究运行时，满足以下条件：

- `planner` 是唯一负责决定工作流结构的组件
- 6 个 agent 保留，但它们是执行身份，不再是硬编码流水线节点
- 每个角色通过 `skills` 执行任务
- 每个 `skill` 通过 `tools` 执行底层动作
- 用户可以通过标准目录结构自由添加新的 `skills`
- `review` 是可选、可插入的，不再是固定必经步骤

## 4. 非目标

第一版不追求以下目标：

- 保留旧 `stage graph` 或旧执行兼容路径
- 保留旧的硬编码 `critic gate`
- 让角色默认直接乱用底层工具
- 允许运行时修改配置
- 支持无约束 shell 或破坏性命令

## 5. 核心产品原则

### 5.1 由 Planner 主导工作流

`planner` 决定：

- 需要哪些角色
- 执行顺序和依赖关系
- 是否需要插入 `reviewer`

### 5.2 角色级路由，工具级执行

- 动态路由发生在角色层
- 真正执行发生在工具层，由 `skill` 向下调用 `tool`
- `planner` 只路由角色和 `skill`，不直接调度原始 `tool`

### 5.3 局部规划

`planner` 只需要规划接下来的一小段工作，通常为 `2-4` 个节点，而不是整个运行周期一次性规划完成。

### 5.4 严格 Contract

运行时必须依赖明确的 `DAG` 与 `artifact contract`，而不是隐式 Python 分支。

### 5.5 用户可扩展 Skill

用户应能在不修改核心运行时代码的情况下，通过标准目录结构添加新的 `skill`。

## 6. 用户故事

- 作为用户，我希望只给出一个研究目标，系统自己决定需要哪些角色。
- 作为用户，我希望系统在任务已经完成时能提前结束，而不是机械地把所有角色都跑一遍。
- 作为用户，我希望 `reviewer` 只在真正需要时出现，而不是固定强插。
- 作为用户，我希望只要按照规范放入一个新的 `skill` 目录，系统就能发现并使用它。
- 作为用户，我希望系统能清楚展示当前正在做什么、产生了什么、为什么重规划。
- 作为用户，我希望系统运行在严格权限约束下，拒绝危险命令和任意删除文件。

## 7. 角色与职责边界

| 角色 | 核心职责 | 明确禁止 |
| --- | --- | --- |
| `conductor` | 拆解任务、组织上下文、定义交接目标 | 不能替代 `writer` 写最终报告 |
| `researcher` | 搜索、收集、阅读、整理研究资料 | 不能充当最终成稿作者 |
| `experimenter` | 设计并执行实验相关技能 | 不能接管通用文献综述主流程 |
| `analyst` | 分析结果、比较证据、解释结果 | 不能负责全局流程决策 |
| `writer` | 根据已有产物生成最终交付物 | 不能偷偷补做上游研究 |
| `reviewer` | 审查产物、指出问题、给出 verdict | 不能直接重写产物 |

## 8. 功能需求

### 8.1 规划能力

- `FR-1` 系统必须包含专门的 `planner`，用于生成局部角色 `DAG`
- `FR-2` `planner` 必须能从 6 个角色中任意选择子集
- `FR-3` `planner` 必须能在某次运行中完全省略 `reviewer`
- `FR-4` `planner` 必须能主动插入 `reviewer`，而不是依赖硬编码流程
- `FR-5` 在不超过最大迭代预算的前提下，`planner` 必须能主动结束任务

### 8.2 执行能力

- `FR-6` `executor` 必须只执行 ready 节点，不能擅自篡改计划
- `FR-7` 角色默认只能通过 `skills` 执行，而不是直接调用底层 `tools`
- `FR-8` 当执行受阻或不确定时，`executor` 必须生成结构化 `observation`
- `FR-9` `planner` 必须能消费这些 `observation` 并输出修订计划

### 8.3 Skill 体系

- `FR-10` 系统必须能从 `skills/<skill_id>/` 自动发现 `skill`
- `FR-11` 每个 `skill` 包必须包含：
  - `skill.yaml`
  - `skill.md`
  - `run.py`
- `FR-12` 每个 `skill` 必须声明适用角色
- `FR-13` `planner` 只能为某个角色分配该角色默认 allowlist 中允许的 `skill`
- `FR-14` 用户必须可以在不修改核心运行时代码的情况下新增 `skill`

### 8.4 Tool 能力

- `FR-15` `skills` 必须可以通过 `ToolGateway` 调用 `MCP tools`
- `FR-16` 工具层必须以 `MCP` 为优先，并在启动时发现已配置 `MCP` 服务暴露的工具能力
- `FR-17` `ToolGateway` 必须按职责拆分，不能重新长成单文件大杂烩
- `FR-18` `planner` 不得直接分配或调用原始 `tool`；具体工具选择发生在 `skill` 执行内部
- `FR-19` 系统必须支持在沙箱和明确允许的远端执行环境中运行代码
- `FR-20` 系统可以联网搜索，并在权限范围内自主选择检索源
- `FR-21` v1 只要求启动时发现工具，不做运行时热插拔或热重载

### 8.5 安全与策略

- `FR-22` 运行时必须有严格的权限策略
- `FR-23` 危险命令必须被阻止
- `FR-24` 任意文件删除必须被阻止
- `FR-25` 运行时配置必须只读
- `FR-26` 系统必须强制执行最大迭代预算

## 9. Skill 包格式

产品层面的 `skill` 包目录结构如下：

```text
skills/
  arxiv_search/
    skill.yaml
    skill.md
    run.py
```

各文件语义如下：

- `skill.yaml`
  - 机器可读说明书
  - 声明 `skill id`、版本、适用角色、输入输出 contract、允许的工具、权限要求
  - `allowed_tools` 应引用通过 `ToolGateway` 暴露的规范化 MCP 工具标识
- `skill.md`
  - 人类可读说明
  - 解释何时使用、预期行为、边界与失败模式
- `run.py`
  - 执行入口
  - 通过允许的工具完成任务，并返回结构化输出

## 10. Planner 插入 Review 的方式

`review` 不再是固定阶段。

相反，review 行为应该通过 `planner` 自己可用的元技能（meta-skills）来驱动，例如：

- `assess_review_need`
- `replan_from_observation`
- `decide_termination`

`planner` 可以在以下情况下考虑插入 `reviewer`：

- 输出不确定性较高
- 证据相互冲突
- 刚完成高成本或高风险节点
- 即将生成重要最终交付物
- 执行过程出现阻塞或退化

## 11. 安全约束

允许的行为：

- 联网搜索
- 自主选择检索源
- 在允许的工作区内读写文件
- 在沙箱内运行代码
- 在显式批准的远端目标上执行任务

禁止的行为：

- 破坏性命令，如 `rm -rf`
- 提权命令，如 `sudo`
- 任意删除文件
- 运行时覆盖配置

## 12. 交互与可视化需求

前端必须如实展示新运行时，而不是继续暗示固定 stage 流程。

必须展示的运行视图：

- 局部 `DAG` 视图
- 角色节点状态
- `skill` 调用日志
- `tool` 调用日志
- `observation` 与 replan 时间线
- `reviewer` 插入节点
- `artifact` 列表与详情
- 权限拒绝信息

UI 不应再暗示固定阶段流水线。

## 13. 成功指标

第一版满足以下条件时视为成功：

- 主运行时不再依赖旧的固定阶段执行链路
- `planner` 真正决定哪些角色执行
- `reviewer` 不再是强制存在
- 用户可以通过标准目录结构新增 `skill`
- 执行失败会生成结构化 `observation`，而不是静默 fallback
- 系统始终运行在预算和权限限制内

## 14. 第一版范围

第一版必须包含：

- `planner` 驱动的局部角色 `DAG`
- 上述 6 个角色
- 基于角色的 `skill allowlist`
- 用户可发现的本地 `skills`
- 启动时的 MCP 工具发现，以及通过模块化 `ToolGateway` 实现的工具调用
- `planner-observation-executor` 重规划闭环
- `planner` 可插入的 `reviewer`
- 基于权限和预算的策略约束

## 15. 明确产品决策

该产品刻意保留“6 个 agent”的模型。

它不会退化成一个单智能体的 `LLM + tools` 通用代理系统。

核心差异是：

- `planner` 负责工作流结构
- 角色负责有边界的执行
- `skills` 负责可复用任务逻辑
- `tools` 负责底层动作执行
